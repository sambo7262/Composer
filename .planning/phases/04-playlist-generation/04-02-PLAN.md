   - Reads Ollama settings via `get_setting(session, "ollama")`

2. **Create `app/services/chat_service.py`**: The orchestration layer implementing D-07 two-phase pipeline and D-08 full conversation context.

   **Session State** (per research Pattern 4):
   ```python
   @dataclass
   class ChatSession:
       session_id: str
       messages: list[dict]  # OpenAI message format [{role, content}]
       current_playlist: list[int]  # Track IDs in order
       track_count: int = 20
   ```
   - Module-level `_sessions: dict[str, ChatSession] = {}` singleton (established pattern in project)
   - `get_or_create_session(session_id: Optional[str]) -> ChatSession`
   - `clear_session(session_id: str)` for "new conversation"

   **System prompt**: Create a concise system prompt (<500 tokens per research) that:
   - Identifies the AI as a music playlist curator
   - Explains feature ranges (energy 0-1, tempo 40-220 BPM, danceability 0-1, valence 0-1 where 0=dark/sad, 1=bright/happy)
   - Instructs the LLM to interpret mood/vibe descriptions into audio feature criteria
   - Notes that genres and artists can be used as additional filters

   **process_message(session_id: str, user_message: str, track_count: int, db_session: Session) -> dict**: Main orchestration function.
   - Get or create ChatSession
   - Append user message to session.messages
   - Get Instructor client from ollama_client
   - **Phase 1 - Mood Interpretation**: Call Instructor with session.messages + system prompt, response_model=FeatureCriteria, via `asyncio.to_thread()`. This produces structured criteria from the mood description (PLAY-02).
   - **Phase 2 - Candidate Filtering**: Call `filter_candidates(db_session, criteria, track_count, candidate_limit=300)` from playlist_engine.
   - **Phase 3 - LLM Curation**: Format candidates with `format_candidates_for_llm()`. Build a second prompt asking the LLM to select exactly `track_count` tracks from the candidates and order them. Call Instructor with response_model=TrackSelection via `asyncio.to_thread()`.
   - **Validate**: Verify all track_ids from TrackSelection exist in the candidates list (T-04-01 threat mitigation). Remove any invalid IDs.
   - Update session.current_playlist with the selected track IDs
   - Append assistant response to session.messages (the explanation)
   - Return dict with: tracks (list of Track objects in playlist order), explanation (str), criteria (FeatureCriteria), session_id

   **Handle smart edits (D-13)**: The process_message function naturally supports "add more like track 3" or "remove the downtempo tracks" because:
   - The system prompt includes the current playlist state
   - The LLM sees full conversation history (D-08)
   - Each response produces a new TrackSelection that replaces the current playlist
   - For refinement messages, include current playlist info in the prompt so the LLM can adjust

   **Handle edge cases**:
   - "not enough tracks": If fewer than 50% of requested tracks score well (score > 0.5), include a warning in the explanation
   - Library exploration (D-10 Flow 3): If the user asks "what jazz do I have?", the LLM should interpret this as a query. The FeatureCriteria will have genres=["jazz"] and wide feature ranges. The explanation should list what was found.
   - Build on existing Plex playlist (D-10 Flow 2): Return a message like "I can't modify existing Plex playlists yet, but I can create a new playlist with a similar vibe. What mood are you going for?" (deferred to Phase 5)

   **Error handling**: Wrap Instructor calls in try/except. On failure (Ollama down, model not loaded, parse failure after retries), return an error dict with a user-friendly message. Use the established `_sanitize_error()` pattern to avoid leaking Plex tokens in error messages.

   **POST /api/chat/message**: Main chat endpoint (per research Pattern 3).
   - Accepts form data: `message` (str), `session_id` (str, optional), `track_count` (int, default 20)
   - Calls `process_message()` from chat_service
   - Returns HTML fragment (two concatenated partials per research):
     a. User message bubble partial (`partials/chat_message.html` with role="user")
     b. AI response partial (`partials/chat_message.html` with role="assistant") -- includes explanation text and, if tracks present, the playlist_card partial
   - On error: return an error message bubble partial
   - Use Jinja2 templates (get_templates() lazy import pattern from pages.py)
   - The response is appended to `#chat-messages` via HTMX `hx-swap="beforeend"`

   **POST /api/chat/remove-track**: Remove a track from current playlist.
   - Accepts form data: `session_id` (str), `track_id` (int)
   - Removes track_id from session.current_playlist
   - Returns empty string (HTMX `hx-swap="outerHTML"` on the `<li>` removes it from DOM)

   **POST /api/chat/reorder**: Reorder tracks in current playlist.
   - Accepts form data: `session_id` (str), `track_id` (int), `position` (int)
   - Moves track_id to new position in session.current_playlist
   - Returns 204 No Content (Alpine.js Sort handles DOM reorder client-side)

   **POST /api/chat/new**: Start a new conversation.
   - Accepts form data: `session_id` (str)
   - Calls `clear_session(session_id)`
   - Returns redirect header or empty state HTML

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User input -> LLM system prompt | Mood descriptions are user-controlled text injected into LLM context |
| LLM output -> database queries | Track IDs from LLM must be validated against actual DB records |
| Client -> API endpoints | Session IDs and track IDs from client requests |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03 | Tampering | chat_service.py | mitigate | Validate all track_ids from TrackSelection exist in the candidates list before using them |
| T-04-04 | Tampering | api_chat.py | mitigate | Validate session_id is a valid UUID format; validate track_id is a positive integer |
| T-04-05 | Information Disclosure | chat_service.py | mitigate | Use _sanitize_error() pattern on Instructor/Ollama errors to avoid leaking Plex tokens or internal paths |
| T-04-06 | Spoofing | chat_service.py | accept | Session IDs are UUID4 (cryptographically random); single-user app so session spoofing is low risk |
